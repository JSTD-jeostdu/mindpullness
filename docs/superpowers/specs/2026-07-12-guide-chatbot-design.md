# mindpullness 안내 챗봇 — 설계

**날짜**: 2026-07-12
**범위**: 앱 이용법을 설명하는 스크립트 기반 FAQ 챗봇 추가 (`chatbot/*.png` 6종 캐릭터 이미지 활용)

## 배경

`chatbot/` 폴더에 청진기를 목에 건 의사 코스프레 캐릭터 6포즈(`1_idle`, `2_clipboard`,
`3_stethoscope`, `4_xray`, `5_syringe`, `6_thumbsup`)가 준비되어 있다. 이 캐릭터를 활용해
앱 사용법을 안내하는 챗봇을 index.html에 추가한다.

CLAUDE.md/PRD의 설계 원칙(서버 비용 0원, API 호출 없음, 단일 파일 유지, 안전 문구 삭제 금지)을
그대로 따른다. 실시간 AI 응답이 아니라 **미리 정의된 질문·답변 트리를 버튼으로 탐색하는 스크립트
FAQ 봇**으로 구현한다.

## 1. 아키텍처

- **파일 구성**: 로직·마크업·스타일은 전부 기존 `index.html`에 추가한다. 새 JS/CSS 파일을 만들지
  않는다. `chatbot/*.png` 6장은 리포지토리에 이미지 애셋으로 추가하고
  `<img src="chatbot/1_idle.png">` 형태의 상대경로로 참조한다.
- **오버레이 재사용**: 새 모달 컴포넌트를 만들지 않고 기존 `#modal-root` / `openModal()` /
  `closeModal()`을 그대로 쓴다. 챗봇 패널은 그 안에 렌더링되는 콘텐츠일 뿐이다. 이 덕분에 다른
  모달이 열려 있을 때나 온보딩 화면(z-index 300)에서 플로팅 버튼이 자연히 가려지며, 별도의
  가시성 제어 코드가 필요 없다.
- **진입점**: `#app` 안에 전역 플로팅 원형 버튼을 추가한다. `1_idle.png`를 원형 아이콘으로 쓰고,
  하단 탭바(`nav`, z-index:50) 위, 화면 우측 하단에 고정한다(`position:fixed; z-index:60`).
  5개 탭 어디서든 노출된다.

## 2. 데이터 구조

질문·답변을 노드 그래프로 정의한다. 각 노드는 캐릭터 포즈, 대사, 다음으로 이어지는 버튼 목록을
가진다. 텍스트 입력창은 없고 버튼 클릭만으로 탐색한다(자유 텍스트 입력 없음 — 스크립트 봇이므로
오타·이해 실패 문제가 발생하지 않도록 하기 위함).

```javascript
const CHATBOT_FAQ = {
  root: {
    pose: "1_idle",
    text: "안녕하세요! 저는 mindpullness 안내 도우미예요. 궁금한 걸 골라주세요.",
    options: [
      {label:"생각기록은 어떻게 써요?", to:"record_1"},
      {label:"클로드 코치가 뭐예요?", to:"coach_1"},
      {label:"28강 배움 코스는요?", to:"learn_1"},
      {label:"호흡·마음챙김 훈련은요?", to:"train_1"},
      {label:"마음체크(PHQ-9/GAD-7)는요?", to:"assess_1"},
      {label:"기기 간 동기화는요?", to:"sync_1"},
      {label:"이 앱은 뭐 하는 앱이에요?", to:"about_1"}
    ]
  },
  // ... 카테고리별 노드 (아래 3절 참고)
};
```

- 모달을 닫았다가 다시 열면 항상 `root`부터 새로 시작한다(대화 기록은 localStorage에 저장하지
  않는 순수 UI 상태 — `S`에는 어떤 필드도 추가하지 않는다).
- 사용자가 버튼을 클릭할 때마다 같은 모달 안에 "질문+답변" 말풍선이 아래로 누적되고, 캐릭터
  이미지가 답변 노드의 포즈로 교체된다. 기존 `.modal`이 `overflow-y:auto`라 자연스럽게 스크롤된다.
- 모든 답변 노드는 마지막에 **"메인 메뉴로"**(`to:"root"`) 버튼을 포함한다.

## 3. 콘텐츠 — 카테고리별 노드 (7개 카테고리 + root)

포즈 매핑:

| 포즈 | 쓰이는 카테고리 |
|---|---|
| `1_idle` | root(첫 인사), 앱 소개 |
| `2_clipboard` | 생각기록 작성법, 클라우드 동기화·데이터 관리 |
| `3_stethoscope` | 마음체크(PHQ-9/GAD-7) |
| `4_xray` | 클로드 코치 연동 |
| `5_syringe` | 호흡·마음챙김 훈련 |
| `6_thumbsup` | 28강 배움 코스 |

각 카테고리는 진입 질문 1개 + 후속 질문 1~3개로 구성한다(최종 문구는 아래 톤 가이드를 따라
구현 단계에서 확정하되, 아래는 실제 반영될 예정 문구다):

**생각기록 (`record_1~4`, pose `2_clipboard`)**
- 진입: "기록 탭 → + 버튼을 누르면 STEP1이 열려요. 상황·자동적 사고·감정·신체반응·행동을 순서대로
  적어요. 상황 칸에는 카메라로 찍히는 사실만 적는 게 포인트예요."
- 후속: STEP2를 꼭 바로 해야 하는지(→ 미완성으로 보존되어 나중에 이어쓰기 가능), 감정 강도는
  어떻게 재는지(→ STEP1/STEP2 슬라이더 0~100, Δ 표시), 거리두기 질문 3종이 뭔지(→ PRD 2.1의
  Q1~Q3 요약)

**클로드 코치 (`coach_1~3`, pose `4_xray`)**
- 진입: 기록 상세의 [🤖 클로드 코치에게 가져가기] 버튼 → 기록+맥락 요약이 프롬프트로 복사되는
  동작 방식
- 후속: 코치 노트가 뭔지(→ 기록에 귀속 저장, 나의 변화에서 모아보기), 주간 회고 프롬프트가 뭔지
  (→ 나의 변화 탭의 복사 버튼)

**28강 배움 코스 (`learn_1~3`, pose `6_thumbsup`)**
- 진입: 4주 28강, 시작일 기준 하루 1강 순차 해금
- 후속: 밀린 강의 몰아봐도 되는지(→ 가능하나 하루 최대 3강 권장), 미션이 뭔지(→ 체크박스 +
  메모, 강의별 저장)

**호흡·마음챙김 훈련 (`train_1~2`, pose `5_syringe`)**
- 진입: 박스 호흡·4-7-8 호흡·생각 흘려보내기·바디 스캔 4종류
- 후속: 시간이 얼마나 걸리는지(→ 호흡 1/3/5분 선택, 나머지 5분 스크립트, 완료 시 로그 누적)

**마음체크 PHQ-9/GAD-7 (`assess_1~3`, pose `3_stethoscope`)**
- 진입: 공개 자가 선별 도구, 진단 도구가 아님
- 후속: 주기가 얼마나 되는지(→ 14일 권장, 홈에서 리마인드), 위기 안내 카드가 뜨는 경우
  (→ **안전 문구는 아래 4절 원문을 그대로 인용**)

**클라우드 동기화·데이터 관리 (`sync_1~3`, pose `2_clipboard`)**
- 진입: 기본은 로컬 저장, Google 로그인 시 개인 Firestore 공간에 미러링
- 후속: 다른 기기에서도 보이는지(→ 같은 계정 로그인 시 필드 병합으로 합쳐짐, 덮어쓰기 아님),
  백업은 어떻게 하는지(→ 나의 변화 → 데이터 관리의 JSON 내보내기/가져오기)

**앱 소개 (`about_1`, pose `1_idle`)**
- **안전 문구는 아래 4절 원문을 그대로 인용**, 새로 창작하지 않는다.

## 4. 안전·톤 가드레일

- 캐릭터는 대화 어디서도 "의사"·"진료"로 지칭하지 않고 **"안내 도우미"**로만 지칭한다.
- 안전 관련 노드(`about_1`, `assess_3`)는 index.html에 이미 있는 문구를 **글자 그대로 재인용**한다
  (신규 문구 창작 금지, CLAUDE.md §9 위반 방지):
  - `about_1`: "이 앱은 교육·자기관리 목적의 도구이며, 의료기기나 심리치료가 아닙니다." +
    "도움이 필요할 땐 · 자살예방 상담전화 109 · 정신건강 상담전화 1577-0199 (24시간)"
    (index.html:1205, 데이터 관리 화면 문구와 동일)
  - `assess_3`: "급성 위기, 정신병적 증상, 자살·자해 관련 문제가 있다면 이 앱보다 정신건강
    전문가의 도움이 우선입니다. (자살예방 상담전화 109, 정신건강 상담전화 1577-0199 · 24시간)"
    (index.html:244, 온보딩 고지와 동일) + "이 검사는 선별 도구일 뿐, 진단이 아닙니다."
    (index.html:1124)
  - PHQ-9 위기 카드 자체(`showAssessResult`의 `.crisis` 블록)는 챗봇과 무관하게 기존 로직 그대로
    유지 — 챗봇은 그 화면으로 유도하는 안내만 제공하고 직접 렌더링하지 않는다.
- 마이크로카피는 PRD 원칙대로 평서형 안내("~해요"), 명령형·평가형 표현을 쓰지 않는다.

## 6. 검증 계획

- `node --check`로 인라인 스크립트 문법 검증
- 브라우저로 7개 카테고리의 모든 클릭 경로를 실제로 밟아보고, 각 답변이 올바른 다음 노드로
  이동하는지, "메인 메뉴로"가 항상 동작하는지 확인
- `about_1`/`assess_3`의 안전 문구 텍스트를 index.html의 원본 문자열과 diff하여 완전히 일치하는지
  확인
- 다른 모달(마법사·마음체크 등)이나 온보딩 화면과 z-index가 충돌하지 않는지 확인
- 콘솔 에러 없이 플로팅 버튼 노출 → 모달 오픈 → 노드 탐색 → 닫기 흐름 확인
